# 코드 규칙 (Code Rules)

이 문서는 `main` 기반 배포와 안전한 롤백을 전제로 하는 개발 규칙 모음입니다.

## 0) 공통 규칙

- 모든 문서, PR 요청문, 커밋 메시지는 **한글**로 작성합니다.
- 문구는 간결하게 작성합니다(필요 이상으로 길게 쓰지 않음).
- 목표 기능이 완성될 때까지 작업을 끝내지 않고 반복 개선합니다.

## 1) 브랜치/PR 운영 규칙

- `main`은 운영 브랜치입니다.
- 기능 개발은 반드시 새 브랜치에서 진행합니다.
- PR은 항상 `main` 대상으로만 생성합니다.
- PR 제목/설명은 아래 형식을 권장:
  - 형식: `<버전> | <기능 요약>`
  - 예시: `v0.3.0 | Graph 동기화 초기화/Delta Sync 연결`
- PR 설명에는 다음 4가지를 포함합니다.
  - 변경 개요(1~2문장)
  - 테스트/커버리지 결과
  - 배포 영향 범위
  - 롤백 포인트

## 2) 커밋 규칙 (rollback-friendly)

- 커밋 단위는 작고 되돌리기 쉬운 단위로 분리합니다.
- 되돌릴 때 특정 기능만 되돌릴 수 있도록 하위 단위 분해를 유지합니다.
- 기본 형식: `타입: 한글 요약`
  - 예: `feat: Graph 동기화 초기 동기화 훅 추가`
- 변경 범위가 넓으면 squash 없이도 단일 의도 단위로 커밋을 쪼갭니다.
- 긴급 복구가 필요하면 `revert:` 접두사로 되돌림 커밋을 생성합니다.

## 3) 코딩 원칙

- 가능한 한 함수형 패러다임을 지향합니다.
  - 순수 함수 우선 사용
  - 상태 변경은 경계(서비스/저장소 계층)에서만 수행
  - 불변 데이터 흐름을 기본으로 유지
- 부수효과는 최소화하고, 가능하면 입력/출력을 명확히 드러냅니다.
- 예외는 조용히 삼키지 않고 호출 계층으로 전달해 일관된 에러 응답으로 처리합니다.
- 중복 로직은 공통 유틸로 분리합니다.

## 4) 테스트 및 품질 기준

- Kent Beck 방식(TDD) 채택
  1. 실패 테스트 먼저 작성
  2. 테스트 통과 코드 작성
  3. 리팩터링
- 핵심 기능은 먼저 테스트로 정합성 정의 후 구현합니다.
- 테스트 커버리지 기준
  - 전체 85% 이상 유지
  - 핵심 기능(동기화/근거/투두/검색)은 PASS 상태를 병합 조건으로 함
- 테스트 파일은 변경 대상 모듈과 같은 계층/폴더에 배치합니다.

## 5) GitHub Actions CI

- CI는 PR 기준으로 필수 실행됩니다.
- 최소 파이프라인
  - 정적 검사(lint)
  - 타입 검사(typecheck)
  - 테스트
  - 커버리지 계산 및 임계치 검사(85%)
  - 빌드/패키지 체크
- 임계치 미달 또는 실패 테스트 1건이라도 있으면 merge는 금지됩니다.

## 6) 변경/문서화 규칙

- PR 본문에는 변경 이유, 검증 상태, 리스크, 다음 액션을 3~5줄 내외로 요약합니다.
- API나 스키마 변경 시 문서를 즉시 갱신합니다.
- `main` 병합 전, 버전 라벨을 명시하지 않으면 승인하지 않습니다.

## 7) 운영 안정성과 롤백

- 기능 플래그 또는 토글 가능한 경로를 우선 적용합니다.
- DB 마이그레이션은 되돌릴 수 있는 순서를 검토한 뒤 반영합니다.
- 배포 직전, "되돌림 단위(모듈/서비스 단위)"를 정의하고 PR에 기록합니다.
- 이상 징후 발생 시: 테스트 실패 → 원인 모듈만 분리 롤백 가능한 커밋 순서를 따라 대응합니다.

## 8) 문서 언어 정책

- 프로젝트 내 문서는 기본 한글 사용.
- 다국어 기술 용어는 괄호 병기 가능(예: `토큰 갱신(refresh)`)
- 코드/식별자는 구현 언어 규칙에 맞게 유지하되, 설명은 한글 우선 사용
